# Baton MVP 이후 필수 구현 범위 메모

> 기준일: 2026-06-23
> 작성 목적: 최종 총괄 판단에서 MVP 이후 작업의 우선순위를 오해하지 않도록 팀장 기준을 명시한다.

## 1. 현재 판단

현재 PURCHASE MVP 흐름은 완성된 상태로 본다.

다만 여기서 만족하는 것이 아니라, 기존에 후순위로 미뤄둔 기능 중 일부는 최종 제출 전까지 최대한 구현하거나 구현 상태를 제출물에 명확히 반영한다.

문서나 회의에서 "추가 기능", "고도화", "MVP 이후"라고 표현하더라도 아래 기능들은 단순 선택 사항이 아니라 사실상 필수 구현 대상으로 관리한다.

## 2. 사실상 필수 구현 대상

| 우선순위 | 기능 | 판단 |
|---|---|---|
| P1 | SWAP 거래 | 반드시 구현 목표로 둔다. Baton의 원래 핵심 방향인 양방향 재능 교환 기능이다. |
| P1 | 관리자 기능 | 반드시 구현 목표로 둔다. 운영/관리 관점의 제출 완성도를 위해 필요하다. |
| P1 | 채팅 | 기본 구현은 완료된 상태로 본다. REST 채팅방/메시지와 WebSocket/STOMP 송수신을 기준으로 시연 가능 여부를 검증하고, Redis Pub/Sub 등은 고도화로 분리한다. |
| P1 | 리뷰/신뢰 점수 | 반드시 구현 목표로 둔다. 거래 이후 신뢰 기반을 만드는 기능이다. |

## 3. 진짜 후순위

아래 기능은 시간이 남으면 진행하되, 위 P1 필수 구현 대상보다 뒤로 둔다.

| 우선순위 | 기능 | 판단 |
|---|---|---|
| P2 | 알림 | 후순위 |
| P2 | 유료 크레딧 충전 | 후순위 |
| P2 | 고도화된 매칭 추천 | 후순위 |
| P2 | 모니터링/성능 고도화 | 후순위. 단, 제출용 최소 로깅/모니터링 근거는 별도 확보 |

## 4. 표현 기준

- PURCHASE는 현재 시연 가능한 최소 안정 흐름이다.
- SWAP, 관리자, 리뷰는 "하면 좋은 기능"이 아니라 최종 완성도를 위해 최대한 구현할 기능이다.
- 채팅은 이미 구현된 기능으로 분류하고, 최종 문서에서는 구현 범위와 시연 가능 범위를 따로 표시한다.
- 시간 관계상 PURCHASE를 MVP로 정리했지만, 원래 서비스 목표에는 양방향 거래, 채팅, 리뷰까지 포함되어 있었다.
- 이후 일정/멘토링/문서 판단에서는 이 기준을 우선 반영한다.
