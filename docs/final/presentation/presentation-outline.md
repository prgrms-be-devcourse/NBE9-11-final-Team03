## 1. 문서/발표 판단

발표의 핵심 메시지는 “돈이 부족한 주니어도 재능을 기반으로 필요한 도움을 얻을 수 있고, Baton은 크레딧과 에스크로로 개인 간 재능 거래의 신뢰 문제를 줄인다”이다.

발표자료는 기능 목록을 나열하기보다 PURCHASE 흐름을 하나의 이야기로 보여준다.

## 3. 슬라이드별 상세 원고

### 1. Baton

화면 문구:

> Baton  
> 크레딧 기반 재능 거래/교환 서비스

발표 멘트:

> Baton은 돈이 부족한 주니어도 자신의 재능을 기반으로 필요한 도움을 얻을 수 있게 하는 서비스입니다. 이번 MVP에서는 양방향 교환보다 먼저 PURCHASE 흐름을 안정적으로 완성하는 데 집중했습니다.

### 2. 문제 정의

화면 문구:

- 주니어는 배울 것과 도움받을 일이 많다.
- 하지만 현금 기반 거래는 부담이 크다.
- 개인 간 거래는 미이행과 이탈 위험이 있다.

발표 멘트:

> 주니어 개발자나 취업 준비생은 코드 리뷰, 디자인 피드백, 포트폴리오 첨삭처럼 서로에게 필요한 도움을 갖고 있습니다. 하지만 현금 거래는 부담스럽고, 개인 간 거래는 신뢰 장치가 약합니다.

### 3. 해결 방식

화면 문구:

> 재능 등록 -> 매칭 제안 -> 크레딧 예치 -> 거래 완료 -> 정산

발표 멘트:

> Baton은 사용자가 재능을 등록하고, 필요한 재능에 매칭 제안을 보낸 뒤, 거래 중 크레딧을 에스크로에 보관합니다. 거래가 완료되면 제공자에게 정산되어 일방적인 이탈 위험을 줄입니다.

### 4. MVP 목표

화면 문구:

| 구분 | 범위 |
|---|---|
| P0 | PURCHASE 핵심 거래 흐름 |
| P1 | 관리자, SWAP, 리뷰, 채팅 |
| P2 | 알림, 유료 충전, 추천 고도화, 모니터링 |

발표 멘트:

> 처음부터 모든 기능을 같은 우선순위로 두기보다, 실제 거래가 성립되는 PURCHASE 흐름을 P0로 잡았습니다. SWAP, 관리자, 리뷰, 채팅은 이후 선택 기능이 아니라 P1 범위로 이어서 관리했습니다.

### 5. 사용자 흐름

화면 문구:

1. 회원가입/로그인
2. 초기 크레딧 지급
3. 재능 등록/조회
4. 매칭 제안 생성/수락
5. 거래 생성/에스크로
6. 완료/정산/이력 기록

발표 멘트:

> 이 흐름이 이번 발표의 핵심입니다. 사용자는 가입 후 크레딧을 받고, 제공자의 재능을 조회하고, 매칭 제안을 보냅니다. 제공자가 수락하면 거래가 생성되고 크레딧은 에스크로에 보관됩니다.

### 6. 시스템 구조

화면 문구:

> Client / Swagger -> Spring Boot API -> Domain Services -> JPA/QueryDSL -> H2/MySQL -> Docker/EC2

발표 멘트:

> 백엔드는 Spring Boot와 Java 21 기반으로 구현했습니다. 도메인은 User, Talent, Matching, Credit, Escrow, Trade로 나누고, 로컬에서는 H2, 운영 구성에서는 MySQL과 Docker 배포를 사용합니다.

### 7. 핵심 API

화면 문구:

| 도메인 | 기능 |
|---|---|
| Auth | 회원가입, 로그인, 토큰 재발급 |
| Talent | 등록, 목록, 검색, 상세 |
| Matching | 추천, 제안 생성, 수락, 거절 |
| Credit | 잔액, 예치, 환불, 이력 구조 |
| Trade | 거래 조회, 취소 |

발표 멘트:

> API는 Swagger에서 도메인별로 확인할 수 있습니다. 시연에서는 기능을 하나씩 보여주기보다, 실제 사용자의 거래 흐름 순서대로 호출해 전체 흐름이 이어지는지 보여드리겠습니다.

### 8. 시연

화면 문구:

> Demo: PURCHASE Flow

발표 멘트:

> 이제 구매자와 제공자 두 사용자를 기준으로 시연하겠습니다. 구매자가 제공자의 재능에 제안하고, 제공자가 수락한 뒤, 거래와 크레딧 상태가 어떻게 이어지는지 확인합니다.

시연 중 체크:

- `buyerId`, `sellerId`
- `talentId`
- `proposalId`
- `tradeId`
- `balance`, `escrowBalance`

### 9. 테스트/품질

화면 문구:

| 항목 | 기준 |
|---|---|
| 테스트 | JUnit |
| 커버리지 | JaCoCo |
| CI | GitHub Actions |
| 배포 | Docker, EC2 |

발표 멘트:

> 품질 관리는 테스트와 커버리지, CI/CD로 진행했습니다. 기존 리포트 기준으로 82개 테스트가 통과했고 Line Coverage는 90.52%였습니다. 다만 최종 발표 전에는 최신 테스트 실패 2건을 정리해 수치를 다시 확정해야 합니다.

### 10. 회고와 확장

화면 문구:

- MVP 범위를 PURCHASE로 좁힌 점
- 에스크로/크레딧 정합성의 중요성
- SWAP, 리뷰, 관리자 기능 구현과 채팅 고도화

발표 멘트:

> 이번 프로젝트에서 가장 중요했던 판단은 PURCHASE 흐름을 먼저 안정화한 것입니다. Baton의 큰 방향은 재능 교환이고, 그 기반 위에서 SWAP과 리뷰, 관리자 기능, 채팅 고도화가 의미 있게 확장될 수 있습니다.

- 슬라이드 수는 10장을 기본으로 한다.
- 한 슬라이드에는 문장 1개와 표/흐름 1개만 둔다.
