## 작성 예시

### 제목

`[Trade] 구매 확정 후 거래 재조회 상태 불일치`

### 1. 문제 상황

| 항목 | 내용 |
| --- | --- |
| 발생 일시 | 2026-06-23 |
| 발견 경로 | Swagger 기준 MVP 통합 테스트 |
| 관련 기능 | Trade / Escrow / Credit |
| 우선순위 | P0 |
| 담당자 |  |
| 상태 | 분석 중 |

### 2. 재현 절차

1. 구매자/제공자 회원가입 및 로그인
2. 제공자 재능 등록
3. 구매자 매칭 제안 생성
4. 제공자 매칭 제안 수락
5. 제공자 결과물 제출
6. 구매자 구매 확정
7. 거래 상세 재조회

### 3. 기대 결과와 실제 결과

| 구분 | 내용 |
| --- | --- |
| 기대 결과 | 구매 확정 후 거래 상태 `COMPLETED`, 에스크로 상태 `RELEASED` |
| 실제 결과 | 구매 확정 응답은 `COMPLETED/RELEASED`, 직후 거래 재조회는 `UNDER_REVIEW/HELD` |
| 영향 범위 | MVP 시연에서 거래 완료 상태 검증 실패 가능 |

### 4. 원인 분석

| 원인 후보 | 확인 내용 | 결과 |
| --- | --- | --- |
| 트랜잭션 커밋 문제 | confirm 처리 이후 Trade/Escrow 저장 여부 확인 |  |
| 조회 DTO 기준 불일치 | confirm 응답 DTO와 detail 조회 DTO 생성 기준 비교 |  |
| 상태 변경 순서 문제 | submission/confirm 상태 전이 순서 확인 |  |

### 5. 해결 방법

| 항목 | 내용 |
| --- | --- |
| 수정 위치 | `TradeService`, `EscrowService`, `CreditService` |
| 해결 방식 | 구매 확정 시 Trade/Escrow/Credit 상태 변경을 하나의 트랜잭션으로 처리하고 조회 기준을 저장 상태로 통일 |
| 테스트 방법 | 구매 확정 후 즉시 거래 상세 재조회 |
| 완료 기준 | confirm 응답과 trade detail 응답이 모두 `COMPLETED/RELEASED` |
