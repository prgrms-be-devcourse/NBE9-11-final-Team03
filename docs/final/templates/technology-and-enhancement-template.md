# 고려 중인 기술 및 고도화 계획 작성 양식

기준일: 2026-06-23
프로젝트: Baton
목적: 최종 발표자료/결과보고서에 포함할 기술 선택 근거와 MVP 이후 고도화 계획 정리

## 1. 작성 원칙

- MVP 필수 기술과 고도화 후보 기술을 분리한다.
- 이미 도입한 기술, 도입 중인 기술, 검토만 한 기술을 구분한다.
- 기술은 이름만 나열하지 않고 `왜 필요한지`, `어디에 쓰는지`, `도입 리스크`, `대체안`을 함께 적는다.
- 고도화 기능은 `구현 완료`, `진행 중`, `계획` 상태를 명확히 표시한다.
- 최종 제출 문서에는 실제 시연 가능한 기능만 `구현 완료`로 표기한다.

### 5.2 작성 예시: 리뷰/신뢰 점수

- 기능명: 리뷰/신뢰 점수
- 우선순위: P1
- 담당자:
- 현재 상태: 계획
- 도입 목적: 거래 완료 이후 사용자 간 신뢰를 축적한다.
- 사용자 가치: 구매자는 신뢰할 수 있는 제공자를 선택할 수 있고, 제공자는 좋은 거래 이력을 쌓을 수 있다.
- 서비스 가치: Baton의 반복 거래와 매칭 품질을 높인다.
- MVP와의 연결: PURCHASE 거래 완료 이후 리뷰 작성이 가능해야 한다.
- 사용 기술: Spring Boot, JPA, QueryDSL
- API 또는 화면: 리뷰 작성, 리뷰 조회, 사용자 신뢰 점수 조회
- 완료 기준: 완료된 거래에 대해 1회 리뷰 작성 가능, 사용자별 평균 점수 조회 가능
- 테스트 기준: 완료되지 않은 거래 리뷰 작성 차단, 중복 리뷰 차단
- 발표 시연 여부: 시간이 있으면 Swagger로 리뷰 작성/조회 시연
- 제출 문서 표기 방식: 구현이 완료되면 P1 구현 기능, 미완성이면 확장 계획
- 리스크: 거래 완료 흐름이 먼저 안정화되어야 한다.
- 남은 작업: 엔티티 설계, API 구현, 테스트 작성

### 작성 예시: S3

- 선택 여부: 선택
- 적용 위치: 재능 첨부파일, 사용자 프로필 이미지, 결과물 파일 확장
- 문제 상황: 파일을 서버 로컬 디스크에 저장하면 배포 환경 변경과 확장에 취약하다.
- 선택 이유: 객체 스토리지 기반으로 파일 저장을 분리하고, 이후 첨부파일 기능을 확장하기 쉽다.
- 기대 효과: 서버와 파일 저장소 분리, 재능 첨부파일 확장 기반 확보
- 트레이드오프: AWS 설정, 권한, 비용, 테스트 환경 분리가 필요하다.
- 대체 기술: 로컬 파일 저장, DB BLOB 저장
- 현재 구현 상태: S3 공통 기반과 결과물 제출용 presigned URL 흐름 도입 완료, 재능 첨부파일 연결은 고도화 범위에서 확인 필요
- 최종 제출 반영 방식: 구현 완료 범위와 확장 기반을 분리해 설명

## 7. 최종 발표에 넣을 요약 문장

> Baton은 PURCHASE MVP를 먼저 안정화한 뒤, 서비스의 신뢰성과 확장성을 높이기 위해 관리자 기능, SWAP 거래, 리뷰/신뢰 점수, 채팅, S3 기반 첨부파일 기능을 고도화 후보로 두었습니다. 기술적으로는 Spring Boot/JPA/QueryDSL/JWT 기반의 핵심 거래 흐름을 유지하면서, 모니터링과 부하테스트, 파일 저장소, 실시간 통신 기술을 단계적으로 검토했습니다.

## 8. 회의에서 결정할 것

- 추가 기능 중 2026-06-28까지 구현 완료로 가져갈 기능

- 채팅을 WebSocket으로 갈지, REST 기반 메시지 저장으로 최소 구현할지
- SWAP을 실제 API로 구현할지, 정책/설계 중심으로 둘지
