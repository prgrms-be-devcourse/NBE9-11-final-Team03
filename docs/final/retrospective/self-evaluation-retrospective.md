# Baton 자체 평가 및 회고

> 문서 버전: v2.0
> 기준일: 2026-07-01

| 항목 | 평가 | 근거 |
| --- | --- | --- |
| 문제 정의 | 명확 | 주니어의 현금·신뢰·진입 장벽을 설정 |
| MVP 범위 | 적절 | SWAP보다 PURCHASE 완주를 우선 |

## 2. 잘한 점

### PURCHASE 우선순위

양방향 SWAP의 정책을 한 번에 해결하기보다 단방향 PURCHASE의 거래·예치·정산을 먼저 안정화했다. 덕분에 Baton의 핵심 가치를 실제 API 흐름으로 검증할 수 있었다.

### 거래 정합성 개선

구매 확정 응답과 DB 재조회 상태가 달라지던 문제를 영속성 컨텍스트 flush 순서에서 찾았다. 벌크 업데이트 전 flush를 보장하고, `EntityManager.clear()` 후 재조회하는 통합 테스트로 `COMPLETED/RELEASED` 저장을 확인했다.

### 여러 단계의 검증

단위·통합 테스트뿐 아니라 실제 HTTP API 흐름과 배포 서버 k6 테스트를 수행했다. 기능의 존재보다 사용자 흐름이 끝까지 이어지는지를 검증했다.

## 3. 아쉬운 점

- 문서 기준일이 달라 구현 상태와 설명이 어긋난 시점이 있었다.
- 작업 브랜치에서 MatchProposal 받은/보낸 목록의 동률 정렬 실패가 발생했으나 최신 DEV에서 `id desc` 보조 정렬로 해결했다.
- API/UI 시연 영상과 CI/CD 성공 로그를 제출 문서에 아직 연결하지 못했다.
- k6 응답 지표는 확보했지만 CPU, 메모리, DB connection 등 서버 자원 모니터링 근거는 부족하다.
- 팀원별 대표 기여를 Jira/PR 링크로 정리하는 작업이 남았다.

## 5. 팀 회고

### 유지할 점

- P0/P1/P2로 범위를 나누고 PURCHASE를 먼저 완성한 점
- 도메인별 책임과 문서 산출물을 남긴 점
- 수동 API와 성능 테스트까지 검증 범위를 확장한 점

### 문제점

- 기능 개발과 문서 갱신 시점이 달라 정합성 점검 비용이 커진 점
- 마감 직전까지 영상, 링크, 테스트 최종 수치가 확정되지 않은 점

### 개선 시도

- 기능 변경 PR에 API/ERD/시연 문서 갱신 여부를 체크한다.
- 발표 전 하루는 기능 추가를 동결하고 회귀 테스트와 리허설에 집중한다.
- 시연 데이터와 ID를 자동 생성하는 Postman 또는 E2E 시나리오를 유지한다.

## 6. 발표용 멘트

> 가장 중요했던 판단은 범위 조절이었습니다. Baton의 장기 목표는 SWAP이지만, 먼저 PURCHASE의 거래·에스크로·정산이 안정적으로 이어져야 한다고 봤습니다. 그 흐름을 실제 API와 부하테스트로 검증한 것이 이번 프로젝트의 핵심 성과입니다. 반면 최종 테스트 한 건과 제출 영상·링크는 마감 전 반드시 닫아야 할 과제로 남아 있습니다.
