## 1. 문서/발표 판단

회고는 변명처럼 보이면 안 된다. “무엇을 목표로 했고, 어디까지 달성했으며, 무엇을 리스크로 관리했는지”를 기준으로 말한다. 특히 PURCHASE MVP와 후순위 기능을 구분한 의사결정을 핵심 회고로 삼는다.

## 2. 자체 평가 요약

| 평가 항목 | 자체 평가 | 근거 |
|---|---|---|
| 문제 정의 | 명확함 | 돈과 신뢰 장벽을 가진 주니어 재능 거래 문제를 설정 |
| MVP 범위 | 적절함 | SWAP보다 PURCHASE 안정화를 우선 |
| 도메인 설계 | 보통 이상 | User, Talent, Matching, Credit, Escrow, Trade로 책임 분리 |
| API 구현 | 보통 이상 | 주요 컨트롤러와 Swagger 기반 확인 가능 |
| 거래 정합성 | 보강됨 | 수락 이후 Trade/Credit/Escrow 연동 구현, 통합 시연 검증 필요 |
| 테스트 | 보강 필요 | 기존 리포트는 양호하나 최신 실행에서 2건 실패 |
| CI/CD | 구성됨 | GitHub Actions, Docker, EC2 배포 workflow 존재 |
| 발표 준비 | 진행 중 | 총괄 문서/구성도/시연 스크립트 작성 |

## 3. 잘한 점

### 3.1 MVP 우선순위 정리

처음부터 SWAP, 채팅, 리뷰, 알림을 모두 구현하려 하지 않고 PURCHASE 흐름을 P0로 분리했다. 이 판단 덕분에 발표의 중심을 “거래가 성립되는가”에 둘 수 있다.

### 3.2 도메인 책임 분리

재능 등록과 검색은 Talent, 제안 생성과 수락은 Matching, 잔액과 예치/환불은 Credit, 보류 상태는 Escrow, 거래 상태는 Trade로 나누었다. 기능이 늘어나도 발표와 코드 설명을 도메인 단위로 할 수 있다.

### 3.3 테스트와 커버리지 관리

기존 품질 리포트 기준으로 User, Talent, Credit, Matching, Escrow, Trade, Global 도메인에 테스트가 작성되어 있다. JaCoCo 수치를 문서화한 점은 제출 자료의 신뢰도를 높인다.

### 3.4 CI/CD 구성

GitHub Actions를 통해 main/dev 브랜치 빌드와 main 브랜치 배포 흐름을 구성했다. Docker Hub와 EC2 배포까지 이어지는 구조가 있어 운영 관점의 설명이 가능하다.

## 4. 아쉬운 점

### 4.1 PURCHASE 전체 자동 연동의 통합 검증 필요

현재 코드 기준으로 `MatchProposalService.acceptMatchProposal`에는 Trade 생성, Credit 에스크로 보류, Escrow 생성이 연결되어 있다. 다만 회원가입부터 구매 확정까지 이어지는 전체 Swagger/Postman 시나리오를 한 번에 검증해야 한다.

### 4.2 회원가입과 초기 크레딧 연결 확인 필요

`CreditService.initializeAccount`는 존재하지만 `AuthService.signup()`에는 Account 생성 TODO가 남아 있다. MVP 성공 기준에는 초기 크레딧 지급이 포함되므로, 자동 지급 흐름을 반드시 연결해야 한다.

### 4.3 테스트 최신 상태 불일치

기존 리포트는 82개 테스트 전체 통과였지만, 2026-06-20 최신 실행에서는 181개 중 2개가 실패했다. 실패 원인은 로그인/토큰 재발급 쿠키의 `Secure` 옵션 검증이다. 최종 발표 전 반드시 테스트 결과를 최신화해야 한다.

### 4.4 문서와 코드 상태 차이

`docs/api/api-spec-v1.md`는 2026-06-22 기준으로 최신 컨트롤러 요약을 반영했다. `docs/erd/erd-v2-review.md`와 테스트 리포트는 아직 2026-06-18 기준 내용이 남아 있으므로 최종 제출 전 최신 코드 기준으로 갱신해야 한다.

## 5. 개선 계획

| 우선순위 | 개선 항목 | 기대 효과 |
|---|---|---|

| P0 | PURCHASE 전체 흐름 통합 검증 | MVP 시연 안정화 |
| P0 | 최신 테스트 2건 해결 및 커버리지 재생성 | 품질 지표 신뢰도 회복 |

## 6. 팀 회고 초안

### Keep

- 기능 우선순위를 P0/P1/P2로 나누어 범위를 관리했다.
- 테스트와 API 문서를 별도 산출물로 남겼다.
- 도메인별 책임을 나누어 기능을 설명하기 쉬운 구조를 만들었다.
- CI/CD와 Docker 배포 구성을 시도했다.

### Problem

- 기획서, API 문서, ERD 문서, 실제 코드의 기준일이 달라 문서 정합성이 흔들릴 수 있다.
- 핵심 PURCHASE 흐름에서 서비스 레벨 구현과 컨트롤러/API 시연 가능 범위가 완전히 일치하는지 추가 검증이 필요하다.
- UI 시연 범위와 배포 링크가 문서에 아직 반영되지 않았다.

### Try

- 최종 발표 전 하루는 기능 추가보다 시연 리허설과 문서 정합성 점검에 집중한다.
- 최종 API 흐름을 Postman collection으로 고정한다.
- 발표자료에는 구현 완료 기능과 후속 확장 기능을 명확히 분리한다.
- 테스트 실패는 발표 전 반드시 해결하거나, 해결하지 못하면 영향 범위를 투명하게 설명한다.

## 7. 발표용 회고 멘트

> 이번 프로젝트에서 가장 중요했던 판단은 범위 조절이었습니다. Baton의 장기 목표는 재능 교환이지만, 먼저 크레딧 기반 PURCHASE 거래가 안정적으로 완성되어야 한다고 봤습니다. 그래서 회원가입, 재능 등록, 매칭 제안, 거래, 에스크로, 정산 흐름을 핵심으로 잡았습니다.

> 아쉬운 점은 문서와 구현 상태가 빠르게 바뀌면서 최종 정합성 검증이 필요해졌다는 점입니다. 특히 매칭 수락 이후 거래와 에스크로, 크레딧 이력이 하나의 흐름으로 완전히 이어지는지는 최종 시연 전 가장 먼저 확인해야 할 항목입니다.

> 이후에는 PURCHASE를 기반으로 SWAP, 리뷰/신뢰 점수, 채팅, 관리자 기능을 확장해 서비스의 실제 사용성을 높이고 싶습니다.
