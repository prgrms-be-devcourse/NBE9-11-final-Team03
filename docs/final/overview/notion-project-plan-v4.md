## 1. 서비스 개요

### 1.1 서비스 소개

Baton은 돈이 부족한 주니어도 자신의 재능을 기반으로 필요한 도움을 얻을 수 있도록 돕는 크레딧 기반 재능 거래/교환 서비스다. 사용자는 자신이 제공할 수 있는 재능을 등록하고, 필요한 재능을 가진 사용자에게 매칭 제안을 보낼 수 있다.

거래에는 현금이 아닌 내부 크레딧을 사용한다. 거래가 시작되면 구매자의 크레딧은 제공자에게 바로 지급되지 않고 에스크로에 보관된다. 거래 완료 또는 구매 확정 조건이 충족되면 제공자에게 정산된다. 이를 통해 개인 간 재능 거래에서 발생할 수 있는 일방 이탈, 미이행, 결과물 미제공 위험을 줄인다.

### 1.2 기획 의도

기존 외주 플랫폼은 이미 검증된 전문가 중심 시장에 가깝다. 의뢰 단가는 높고, 거래를 시작하려면 포트폴리오와 평점이 어느 정도 갖춰져 있어야 한다. 반면 주니어 개발자, 디자이너, 기획자, 취업 준비생은 서로에게 필요한 도움을 줄 수 있지만 현금 부담과 신뢰 부족 때문에 거래를 시작하기 어렵다.

Baton은 이 문제를 내부 크레딧과 에스크로 구조로 해결한다. 사용자는 자신의 재능을 크레딧 가치로 등록하고, 재능 제공을 통해 얻은 크레딧으로 다른 사람의 재능을 요청할 수 있다. 선의에만 의존하지 않고 시스템이 거래 상태와 크레딧 흐름을 관리하는 것이 Baton의 핵심 방향이다.

### 1.3 MVP 방향

Baton의 최종 지향점은 양방향 재능 교환까지 가능한 플랫폼이다. 다만 이번 MVP에서는 정책 복잡도가 높은 SWAP보다, 단방향 구매 거래인 PURCHASE 흐름을 먼저 안정화한다.

MVP의 핵심은 다음 흐름이 Swagger 또는 Postman에서 순서대로 시연 가능해야 한다는 점이다.

1. 회원가입/로그인
2. 초기 크레딧 지급
3. 재능 등록/조회
4. 매칭 제안 생성
5. 매칭 제안 수락
6. PURCHASE 거래 생성
7. 구매자 크레딧 차감
8. 에스크로 보관
9. 거래 상태 조회
10. 거래 완료/구매 확정
11. 제공자 크레딧 정산
12. 크레딧 변동 내역 기록

SWAP, 리뷰/신뢰 점수, 채팅, 관리자 기능은 포기 기능이 아니라 PURCHASE 안정화 이후 이어서 완성도를 높일 P1 범위다. 특히 채팅은 기본 구현이 완료된 상태이므로, 최종 제출 전에는 구현 범위와 시연 가능 범위를 검증한다.

## 2. 문제 정의

### 2.1 현금 부담

주니어는 성장 과정에서 코드 리뷰, 디자인 피드백, 포트폴리오 첨삭, 면접 준비처럼 다양한 도움이 필요하다. 그러나 기존 서비스는 현금 지불을 전제로 하며, 단가와 수수료가 부담으로 작용한다.

### 2.2 신뢰 부족

커뮤니티나 오픈채팅을 통한 개인 간 재능 교환은 안전장치가 부족하다. 한쪽이 결과물을 받고 잠수하거나, 약속된 품질을 이행하지 않거나, 거래 도중 이탈해도 대응하기 어렵다.

### 2.3 주니어 진입 장벽

전문가 중심 플랫폼에서는 완성도 높은 포트폴리오와 거래 이력이 없으면 제공자로서 거래를 시작하기 어렵다. Baton은 작은 재능도 등록하고 거래할 수 있는 주니어 친화 구조를 목표로 한다.

## 3. 해결 방식

Baton은 다음 방식으로 문제를 해결한다.

- 현금 대신 내부 크레딧을 사용해 시작 비용을 낮춘다.
- 거래 중 크레딧을 에스크로에 보관해 미이행 위험을 줄인다.
- 거래 상태, 크레딧 보류, 정산 이력을 시스템이 관리한다.
- PURCHASE 거래를 먼저 안정화하고, 이후 SWAP으로 확장한다.

## 8. 주요 기능

### 8.1 회원/인증

MVP에서는 이메일 기반 회원가입, 로그인, JWT 발급, refresh token 재발급을 우선한다. 소셜 로그인과 휴대폰 본인인증은 후순위로 둔다.

주요 기능:

- 이메일 기반 회원가입
- 로그인
- JWT access token 발급
- refresh token 기반 재발급
- 사용자 기본 프로필 관리

후순위:

- 소셜 로그인
- 휴대폰 본인인증
- 제재 유저 우회 가입 방지 고도화

### 8.2 재능

사용자는 자신이 제공할 수 있는 재능을 등록하고, 다른 사용자의 재능을 목록/검색/상세 조회로 탐색한다. 재능에는 카테고리, 제목, 설명, 예상 소요 시간, 크레딧 가격이 포함된다.

주요 기능:

- 재능 등록
- 재능 수정
- 재능 삭제
- 재능 목록 조회
- 재능 검색 및 필터링
- 재능 상세 조회
- 커서 기반 페이지네이션

후순위:

- 카테고리별 권장 가격
- 첨부파일 고도화
- 신고 기능

### 8.3 매칭/MatchProposal

구매자는 원하는 재능에 대해 매칭 제안을 생성하고, 제공자는 제안을 수락하거나 거절할 수 있다. PURCHASE에서는 요청자의 재능이 없을 수 있으므로 requesterTalentId는 null일 수 있다.

| 상태 | 의미 |
|---|---|
| REQUESTED | 매칭 제안 생성 |
| ACCEPTED | 제공자가 수락 |
| REJECTED | 제공자가 거절 |
| CANCELLED | 요청자가 취소 |

주요 기능:

- 추천 상대 조회
- 추천 상대 상세 조회
- 매칭 제안 생성
- 매칭 제안 수락
- 매칭 제안 거절

후순위:

- 받은 제안/보낸 제안 목록
- 제안 취소 API
- 추천 알고리즘 고도화
- SWAP 매칭 그룹

### 8.4 거래/Trade

MVP의 거래는 PURCHASE를 기준으로 한다. 매칭 제안이 수락되면 거래가 생성되고, 구매자의 크레딧이 차감되어 에스크로에 보관된다.

주요 기능:

- PURCHASE 거래 생성
- 거래 상태 조회
- 거래 취소
- 결과물 제출
- 구매 확정
- 거래 완료

검증 필요:

- 수정 코드가 반영된 실행 서버에서 구매 확정 후 거래 상세 재조회 상태가 confirm 응답과 일치하는지 API 재검증
- Swagger/Postman 시연 중 사용할 `tradeId` 확인 방식 정리

### 8.5 크레딧/Credit

Baton은 내부 가상 크레딧을 기반으로 거래한다. CreditAccount는 사용 가능한 balance와 거래 중 보류된 escrowBalance를 분리해 관리한다.

| 필드 | 의미 |
|---|---|
| balance | 즉시 사용할 수 있는 크레딧 |
| escrowBalance | 거래 중 에스크로에 보류된 크레딧 |

주요 기능:

- 회원가입 시 초기 크레딧 지급
- 크레딧 잔액 조회
- 거래 시 크레딧 차감
- 에스크로 보류
- 거래 취소 시 환불
- 거래 완료 시 제공자 정산
- 크레딧 변동 이력 기록

검증 필요:

- WELCOME, ESCROW_HOLD, ESCROW_RELEASE 이력 조회 결과 최종 시연 데이터 확보
- 동시 요청 시 중복 차감 방지
- 구매자 `ESCROW_RELEASE` 이력의 amount 표기 의미 정리

### 8.6 에스크로/Escrow

Baton은 외부 PG 에스크로가 아니라 내부 크레딧 기반 자체 에스크로 테이블을 사용한다. 거래 중 구매자의 크레딧은 제공자에게 즉시 지급되지 않고 HELD 상태로 보관된다.

주요 기능:

- 에스크로 생성
- 크레딧 보류
- 거래 취소 시 환불
- 거래 완료 시 정산
- 구매 확정 만료일 관리

후순위:

- 분쟁 상태
- 관리자 중재
- 자동 정산 스케줄러

### 9.1 기능별 진행 상태와 일정 준수 기준

중간 피드백 이후 진행 상태는 단순 기능 구현 여부가 아니라 PURCHASE MVP 완주 가능 여부를 기준으로 관리한다.

| 도메인 | 기능 | 현재 판단 | 일정/검증 기준 |
|---|---|---|---|
| Auth/User | 회원가입, 로그인, 토큰 재발급 | 구현 | Swagger에서 회원가입/로그인 가능 |
| Credit | 회원가입 후 초기 크레딧 지급 | 구현/검증 완료 | 회원가입 직후 CreditAccount와 WELCOME 이력 생성 |
| Talent | 재능 등록/조회/검색 | 구현 | 제공자 재능 등록 후 구매자가 조회 가능 |
| Matching | 제안 생성/수락/거절 | 구현 | 수락 시 거래 흐름으로 진입 |
| Trade/Credit/Escrow | 수락 후 거래 생성/크레딧 보류/에스크로 생성 | 구현 | Trade 생성, balance 감소, escrowBalance 증가, Escrow HELD |
| Trade Submission | 결과물 제출/조회 | 구현 | 제출 후 Trade UNDER_REVIEW |
| Settlement | 구매 확정/정산 | 코드 수정/통합 테스트 완료 | confirm 후 DB 재조회 `COMPLETED/RELEASED` 확인, 서버 API 재검증 필요 |
| History | 크레딧 변동 이력 | 구현/검증 완료 | `/api/v1/credit/transactions`로 WELCOME, ESCROW_HOLD, ESCROW_RELEASE 조회 |
| Demo | Swagger/Postman 전체 시연 | 정상 흐름 검증 완료 | 회원가입부터 정산까지 순서대로 재현, 제출용 영상 녹화 필요 |

### 9.2 PURCHASE 상태 전이

| 도메인 | 상태 전이 |
|---|---|
| MatchProposal | `REQUESTED -> ACCEPTED`, `REQUESTED -> REJECTED`, `REQUESTED -> CANCELLED` |
| Trade | `IN_PROGRESS -> UNDER_REVIEW -> COMPLETED`, `IN_PROGRESS` 취소 불가, 필요 시 `DISPUTED` |
| Escrow | `HELD -> RELEASED`, `HELD -> REFUNDED`, 필요 시 `FROZEN` |
| CreditTransaction | `WELCOME`, `ESCROW_HOLD`, `ESCROW_RELEASE`, `REFUND` |

상태 변경은 이전 상태를 검증한 뒤 다음 상태로만 이동하도록 관리한다. 특히 구매 확정은 결과물 제출 이후에만 가능해야 하며, 중복 확정으로 인한 중복 정산이 발생하지 않도록 Trade와 Escrow 상태를 함께 확인한다.

### 9.3 도메인 책임 경계

| 도메인 | 책임 | 책임 밖 |
|---|---|---|
| Matching | 제안 생성, 수락, 거절, 제안 상태 관리 | 잔액 계산, 정산 |
| Trade | 거래 생성, 거래 상태 전이, 결과물 제출/구매 확정 흐름 | 크레딧 잔액 직접 조작 |
| Credit | 사용 가능 크레딧, 에스크로 보류 크레딧, 변동 이력 관리 | 거래 상태 판단 |
| Escrow | 보류 금액의 HELD/RELEASED/REFUNDED 상태 관리 | 사용자 인증, 매칭 정책 |
| Talent | 재능 등록/조회/첨부파일 | 거래 정산 |

### 9.4 동시성 및 트랜잭션 전략

크레딧 차감, 에스크로 생성, 거래 생성은 하나의 PURCHASE 흐름으로 묶어 정합성을 관리한다.

| 상황 | 위험 | 대응 방향 |
|---|---|---|
| 같은 매칭 제안 중복 수락 | Trade 중복 생성 | MatchProposal 상태 검증 및 `Idempotency-Key` 사용 |
| 구매자 크레딧 중복 차감 | balance 음수 또는 중복 hold | CreditAccount update 조건 검증, 트랜잭션 처리 |
| 거래 생성 후 에스크로 실패 | Trade/Credit/Escrow 불일치 | `@Transactional`로 전체 롤백 |
| 구매 확정 중복 호출 | 제공자 중복 정산 | Trade 상태와 Escrow 상태 검증 |
| SWAP 확장 | 양방향 거래 불일치 | TradeGroup과 방향별 idempotency key 검토 |

## 10. SWAP 확장 방향

SWAP은 Baton의 장기 차별화 방향이면서 최종 제출 전까지 우선 구현을 검토할 P1 범위다. PURCHASE를 먼저 안정화한 뒤, 두 개의 단방향 거래를 하나의 TradeGroup으로 묶어 SWAP을 구현한다.

### 10.1 구조 초안

| 구성 | 설명 |
|---|---|
| TradeGroup | 양방향 교환 묶음 |
| Trade A | 사용자 A가 사용자 B에게 제공하는 거래 |
| Trade B | 사용자 B가 사용자 A에게 제공하는 거래 |
| Escrow A/B | 양쪽 거래 각각의 보류/정산 상태 |

### 10.2 미결 정책

- 두 재능의 creditPrice 차이를 허용할지 여부
- 차액을 크레딧으로 추가 예치할지 여부
- 한쪽만 결과물을 제출한 경우 취소/정산 기준
- 한쪽만 구매 확정한 경우 자동 완료 여부
- 분쟁 발생 시 개별 거래 단위로 처리할지 TradeGroup 단위로 처리할지

## 17. 발표용 요약

Baton은 주니어가 가진 재능을 크레딧으로 연결해 필요한 도움을 얻을 수 있게 하는 서비스다. 이번 MVP에서는 실제 거래가 성립되는 PURCHASE 흐름을 우선 완성하는 데 집중했다. 구매자가 매칭 제안을 보내고 제공자가 수락하면 거래가 생성되며, 구매자의 크레딧은 바로 지급되지 않고 에스크로에 보관된다. 이 구조를 통해 개인 간 재능 거래에서 발생할 수 있는 미이행과 일방 이탈 위험을 줄이고, 이후 SWAP, 리뷰/신뢰 점수, 관리자 기능, 채팅 고도화로 확장할 수 있는 기반을 만든다.
