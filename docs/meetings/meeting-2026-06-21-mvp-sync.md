# 2026-06-21 Baton 회의 진행 문서

## 0. 회의 목적

오늘은 2026년 6월 21일 일요일이다. 오늘 회의의 목적은 단순 진행 공유가 아니라, 남은 기간 동안 팀이 어떤 기준으로 움직일지 확정하는 것이다.

회의가 끝났을 때 아래 4가지가 정해져 있어야 한다.

- PURCHASE MVP 흐름에서 아직 끊긴 지점
- 끊긴 지점별 최종 책임자
- 고도화/추가 기능의 우선순위와 담당자
- Swagger/Postman 시연 가능 여부를 기준으로 한 완료 판단 방식

회의 기준:

> 기능별 구현 완료가 아니라, 사용자가 실제로 `회원가입 -> 재능 등록 -> 매칭 제안 -> 수락 -> 거래 생성 -> 에스크로 -> 구매 확정 -> 정산`까지 따라갈 수 있는지를 완료 기준으로 본다.

## 1. 현재 총괄 판단

원격 `dev`, Jira, 정책 문서 기준으로 현재 상태는 아래와 같다.

- Swagger API 설명 PR은 머지됨.
- S3 공통 코드도 이미 머지됨.
- 재능, 매칭, 거래, 크레딧 기능 조각은 상당 부분 구현됨.
- 하지만 PURCHASE MVP의 핵심 연결부는 아직 부족함.
- Notion 정책 문서는 운영 정책까지 넓게 잡혀 있으므로, MVP 적용 정책과 후속 운영 정책을 분리해야 함.

가장 큰 리스크:

- `BATON-48`: 회원가입 시 초기 크레딧 계좌 생성
- `BATON-79`: 매칭 수락 후 거래/에스크로/크레딧 연결
- `BATON-72`: 구매 확정 및 제공자 정산

## 3. 회의 시작 멘트

회의 시작 때 아래처럼 말한다.

> 오늘 회의는 진행 공유도 하지만, 핵심은 MVP 흐름 연결입니다.

> 다만 회원가입 후 초기 크레딧, 매칭 수락 후 거래/에스크로/크레딧 연결, 구매 확정/정산이 아직 PURCHASE 흐름에서 끊겨 있습니다.

> 추가 기능 개발은 반드시 하되, MVP 안정화와 병렬 트랙으로 나눠서 진행하겠습니다.

## 4. 역할 분할 원칙

기존처럼 기능별로만 나누면 `각자 기능은 끝났는데 전체 흐름은 안 되는 상황`이 생긴다. 그래서 아래 3층 구조로 나눈다.

### 4.1 도메인 오너

| 담당자 | 기존 중심 역할 | 계속 맡을 것 |
|---|---|---|
| 이유진 | 회원/Auth | 회원가입, 로그인, 토큰, 회원 상태 |
| 박재현 | Talent | 재능 등록/조회/검색/첨부파일 |
| 이인희 | Matching | 추천, 매칭 제안 생성/수락/거절 |
| 최윤서 | Trade/Credit/Escrow/S3 | 거래, 에스크로, 크레딧, S3 |
| 남진우 | 총괄/문서/QA | 일정, Jira/PR, 시연, 발표 흐름 |

### 4.2 MVP 흐름 책임자

| MVP 흐름 | 책임자 | 협업자 | 완료 기준 |
|---|---|---|---|
| 회원가입 후 초기 크레딧 지급 | 이유진 | 최윤서 | 회원가입 직후 `CreditAccount`와 WELCOME 거래 내역 생성 |
| 매칭 수락 후 PURCHASE 거래 생성 | 이인희 | 최윤서 | 제안 수락 시 `Trade`가 생성되고 조회 가능 |
| 구매자 크레딧 에스크로 보류 | 최윤서 | 이인희 | 수락 후 구매자 `balance` 감소, `escrowBalance` 증가, HOLD 내역 기록 |
| 거래 완료/구매 확정 | 최윤서 | 이인희 | 구매 확정 API 호출 시 거래 상태 `COMPLETED` |
| 제공자 정산 | 최윤서 | 이유진 | Escrow `RELEASED`, 제공자 크레딧 증가, RELEASE 내역 기록 |
| 전체 시연 시나리오 | 남진우 | 전원 | Swagger/Postman에서 a~k 흐름 재현 |

### 4.3 고도화 병렬 트랙

| 고도화 후보 | 추천 담당 | 추천 범위 | 우선순위 |
|---|---|---|---|
| 리뷰/신뢰 점수 | 이유진 + 최윤서 | 거래 완료 후 리뷰 작성, 평점/신뢰 점수 반영 | 1 |
| 관리자 기능 | 남진우 + 최윤서 | 사용자/거래/신고 조회, 상태 관리 | 2 |
| 채팅 | 이인희 | 실시간 전 단순 메시지 저장/조회부터 | 3 |
| SWAP | 이인희 + 최윤서 | 설계/상태 확장 중심, PURCHASE와 분리 | 4 |
| 재능 첨부파일 | 박재현 | S3 기반 첨부파일 등록/조회 | 진행 중 |

추천 결정:

- 발표에 보여줄 고도화 1순위는 `리뷰/신뢰 점수`가 가장 좋다.
- 이미 진행 중인 `재능 첨부파일`은 박재현이 계속 진행하되, MVP 흐름에 영향 없게 분리한다.
- 관리자 기능은 시간이 남으면 운영 정책과 연결해서 발표 보조 기능으로 가져간다.

## 7. P0 작업 상세 정의

### BATON-48: 회원가입시 계좌 생성

최소 구현:

- 회원가입 성공 후 `CreditService.initializeAccount(userId)` 호출
- `CreditAccount` 생성
- 초기 지급 크레딧 반영
- `CreditTransactionType.WELCOME` 내역 생성

완료 확인:

- 회원가입 API 호출
- 생성된 userId로 크레딧 잔액 조회
- `balance = initialCreditAmount`, `escrowBalance = 0`

주의:

- 회원 생성과 계좌 생성은 같은 트랜잭션으로 묶는 것이 좋음
- 중복 호출 시 계좌 중복 생성 방지

### BATON-79: 거래 제안 흐름 검증 및 연결

최소 구현:

- `acceptMatchProposal()`에서 provider 권한 검증
- providerTalent 조회
- 구매자/requester와 제공자/provider 확정
- `Trade.create(..., TradeType.PURCHASE)` 호출
- `Escrow.create(...)` 호출
- `CreditService.holdForEscrow(...)` 호출
- 모두 성공한 뒤 `matchProposal.accept()`

완료 확인:

- 매칭 제안 수락 API 호출
- Trade 생성 확인
- Escrow HELD 확인
- 구매자 balance 감소, escrowBalance 증가
- CreditTransaction ESCROW_HOLD 기록

주의:

- 실패하면 ACCEPTED로 바뀌면 안 됨
- idempotencyKey는 `"MATCH-ACCEPT-" + proposalId` 또는 `"TRADE-HOLD-" + tradeId`처럼 재현 가능하게 잡기

### BATON-72: 결과물 확인 및 구매 확정

최소 구현:

- 구매자 또는 거래 참여자 권한 검증
- Trade 상태 `IN_PROGRESS -> COMPLETED`
- Escrow 상태 `HELD -> RELEASED`
- 구매자 escrowBalance 감소
- 제공자 balance 증가
- CreditTransaction ESCROW_RELEASE 기록

완료 확인:

- 거래 조회에서 `COMPLETED`, `RELEASED` 확인
- 제공자 잔액 증가 확인
- 구매자 escrowBalance 감소 확인
- 정산 내역 기록 확인

주의:

- MVP에서는 Auto-Release, Dispute, 수정 요청은 제외
- `UNDER_REVIEW`가 꼭 필요하지 않다면 구매 확정 API 하나로 단순화

## 8. 정책 문서 싱크 정리

Notion v2 정책 문서는 운영 정책까지 넓게 포함한다. 최종 제출 문서에서는 아래처럼 분리한다.

| 정책 항목 | 현재 문서 | MVP 반영 여부 | 정리 방향 |
|---|---|---|---|
| 초기 크레딧 어뷰징 방지 | rate limit, 다중 계정 방지 | 일부만 반영 | MVP는 이메일 중복 기준 1회 지급 |
| 재능 승인제 | PENDING/APPROVED/REJECTED/HIDDEN | 미반영 | 관리자 고도화로 분리 |
| 거래 스냅샷 | trade_snapshots | 일부만 반영 | MVP는 거래 생성 시 가격 고정 저장 |
| 매칭 타임아웃 | TIMEOUT_CANCELLED | 미반영 | 후속 |
| 노쇼 패널티 | penalty_score | 미반영 | 후속 |
| Auto-Release | 자동 구매 확정 | 미반영 | 후속 |
| Dispute | 분쟁/증빙 | 미반영 | 후속 |
| 관리자 감사 로그 | admin_audit_logs | 미반영 | 관리자 고도화 |
| 채팅 보존 | chat_messages, 증빙 | 미반영 | 채팅 고도화 |

회의 결론으로 남길 문장:

> 정책 문서는 유지하되, 최종 발표에서는 MVP 적용 정책과 후속 운영 정책을 구분한다.

## 12. 회의 운영 팁

논의가 길어지면 아래 기준으로 끊는다.

- MVP 시연을 막는가? 그러면 P0
- 발표에서 보여줄 기능인가? 그러면 P1
- 운영상 있으면 좋은가? 그러면 P2
- 정책에는 있지만 구현이 큰가? 그러면 후속 정책

팀장이 마지막에 정리할 말:

> 오늘 결정은 “MVP를 먼저 끝내고 고도화는 포기한다”가 아닙니다.  
> MVP 흐름을 기준선으로 고정하고, 고도화는 병렬로 가져가자는 것입니다.  
> 그래서 오늘부터는 기능별 완료가 아니라 Swagger/Postman에서 이어지는 흐름을 완료 기준으로 보겠습니다.
