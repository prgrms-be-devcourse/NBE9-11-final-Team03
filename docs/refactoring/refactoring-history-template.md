## 1. 작성 기준

이 문서는 단순 수정 기록이 아니라, **프로젝트 품질을 높인 의미 있는 변경**만 기록한다.

기록 대상:

| 기록 대상 | 예시 |
|---|---|
| 구조 개선 | 서비스 책임 분리, 검증 로직 분리 |
| 설계 개선 | `balance` / `escrowBalance` 분리 |
| 공통화 | `ApiResponse`, `ErrorCode`, ExceptionHandler 정리 |
| 테스트 보강 | Controller/Service 테스트 추가 |
| 문서 반영 | API 명세서, ERD, WBS 최신화 |
| 후속 분리 | Trade/Escrow/CreditTransaction 연동 티켓 분리 |

제외 대상:

| 제외 대상 | 이유 |
|---|---|
| 단순 오타 수정 | 품질 개선 이력으로 보기 어려움 |
| import 정리 | 별도 기록 불필요 |
| 포맷팅 변경 | 기능/구조 영향 없음 |
| 의미 없는 변수명 변경 | 정책/구조 변화가 없으면 제외 |
| 개인 작업 메모 | Jira 댓글 또는 PR 코멘트로 충분 |

## RF-001. CreditAccount 잔액 구조 분리

| 항목 | 내용 |
|---|---|
| 구분 | 설계 개선 |
| 도메인 | Credit |
| 처리 상태 | 완료 |
| 관련 PR |  |
| 관련 Jira |  |
| 담당자 |  |
| 반영 브랜치 | `dev` |
| 반영일 | 2026-06-18 |

### 1. 개선 전 문제

```text
사용 가능한 크레딧과 거래 중 에스크로에 묶인 크레딧을 하나의 잔액으로만 관리하면,
구매자가 실제로 사용할 수 있는 금액과 예치 중인 금액을 구분하기 어렵다.
```

### 2. 개선 내용

```text
CreditAccount에 balance와 escrowBalance를 분리했다.
balance는 사용 가능 크레딧, escrowBalance는 거래 중 에스크로로 보류된 크레딧을 의미한다.
```

### 3. 개선 효과

```text
PURCHASE 거래에서 크레딧 예치, 환불, 정산 흐름을 명확하게 표현할 수 있다.
API 명세서와 ERD에서도 사용 가능 잔액과 예치 잔액을 구분해 설명할 수 있다.
```

### 4. 영향 범위

| 영향 대상 | 내용 |
|---|---|
| API | 크레딧 잔액 조회 응답에 `balance`, `escrowBalance` 포함 |
| DB | `credit_account.balance`, `credit_account.escrow_balance` 사용 |
| 테스트 | 크레딧 잔액 조회 테스트 필요 |
| 문서 | API 명세서, ERD v2 반영 필요 |
| 후속 작업 | MatchProposal 수락 시 에스크로 보류 연동 필요 |

### 5. 후속 작업

| 항목 | 내용 |
|---|---|
| 후속 Jira 필요 여부 | 필요 |
| 후속 티켓 | `[Follow-up] MatchProposal 수락 시 Trade/Escrow/Credit 연동` |
| 남은 리스크 | 동시 요청 시 잔액 정합성 검증 필요 |
| 보완 예정 시점 | MVP PURCHASE 흐름 구현 단계 |
